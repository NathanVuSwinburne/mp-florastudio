## What this product is

FloraStudio is a **B2C** app for home gardeners. Target user: **non-technical,
often older** (the founder's mum is the archetype). It has two subsystems:

1. **Garden design studio** — screenshot your space, drag/drop realistic plants
   & decor, get a shopping list. (Already a working frontend prototype.)
2. **Plant health monitoring** — photograph a plant, get a health score + plain
   advice. (This is what we are now building — Phase 2.)

The guiding UX principle everywhere: **dead-simple for non-tech users.** Every
technical decision must reduce, not add, friction for that audience.

## The commercial goal

This is a product that must **make money from users**, not a demo. Decisions are
judged on two axes: (a) does it serve the non-tech B2C user, and (b) does it
improve unit economics / willingness to pay. Cost structure matters — see the
"cost architecture" below and `docs/DECISIONS.md`.

## Health-monitoring architecture (decided — do not relitigate)

```
photo → [Model A: on-device CNN]  → {healthy|stressed|diseased} + confidence
      → confidence gate (< 0.6 ⇒ "uncertain", honest "not sure" UX)
      → [Plant Carer Agent (LLM, backend)] tool-calls as needed:
           run_health_model · view_image · get_care_recommendation ·
           get_weather · get_plant_profile
      → diagnosis + plain-language advice + concrete actions
      → user acts, taps existing care buttons → health responds
      → [data flywheel] stores photo + feedback + (action→outcome) → retrain
```

Key locked decisions (rationale in `docs/DECISIONS.md`):
- **Custom ML is image-based, not sensor-based.** Users won't log sensor data;
  they will take photos.
- **Diagnosis runs on-device** (TF.js). Free/instant/private/offline for the
  common case; backend only for the agent + flywheel storage.
- **LLM is a tool-calling "Plant Carer Agent"**, not a fixed pipeline. It writes
  advice and, on low confidence, inspects the image itself (the fallback).
- **Care recommender starts rules-based** (species care table), becomes ML only
  once the flywheel has (action→outcome) data. Do not train it yet.
- **The flywheel is the moat**, not the launch-day model.

## Phasing

- **Phase 2a (current plan):** train Model A, run it in-browser, replace the
  mock photo check. Ships value alone.
- **Phase 2b:** Plant Carer Agent + backend + tools.
- **Phase 2c:** data flywheel (storage, consent, feedback, retraining).

## Honest limitation for autonomous runs

A headless agent **cannot** download the multi-GB PlantVillage/PlantDoc datasets
or train on a GPU. Therefore, overnight, the agent:
- builds **all** Track A code + Track B code and keeps it tested,
- runs the app end-to-end against the **stub model** (`public/models/health-v1/`,
  generated by `ml/florastudio_ml/make_stub_model.py`),
- leaves the real GPU steps (download → train → evaluate → export) as clearly
  marked TODOs for a human. **Never fabricate a "trained" model or fake metrics.**

## Non-goals for now (YAGNI)

Species-ID model, sensor/IoT hardware, trained care recommender, checkout/
payments, and the garden-design subsystem changes — all out of the health plan.
